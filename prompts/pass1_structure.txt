FIELD SPECIFICATIONS
═══════════════════════════════════════════════════════════════════════

## sections (required)

Each section represents a chapter section or subsection.

- **unit_id**: Hierarchical identifier using dot notation (e.g., "1.1", "1.2.3")
- **title**: Section heading as it appears in text (or descriptive title if none)
- **level**: Nesting depth (1 = top-level, 2 = subsection, 3 = sub-subsection)
- **parent_unit_id**: Parent section's unit_id (null for top-level sections)

Extract ALL sections and subsections. Typical chapter: 5-15 top-level sections.

## summary (required)

- One well-crafted paragraph (3-6 sentences)
- Captures main thesis, scope, and key topics
- Be substantive, not generic ("This chapter discusses..." is too vague)

## key_entities (required, can be empty)

People, organizations, concepts, events central to the chapter.

- **name**: Entity name as it appears in text
- **type**: One of: person, organization, concept, event, theory, movement, technology, place, document

Include only entities discussed substantively (not just mentioned). Typically 5-20 per chapter.

## keywords (required, can be empty)

Domain-specific terminology. 10-30 terms per chapter.

- Technical jargon and field-specific terms
- Use lowercase unless proper noun
- Prefer specific over generic

QUALITY CHECKLIST
═══════════════════════════════════════════════════════════════════════

✓ All section hierarchies are correct (parent_unit_id matches)
✓ unit_id numbering is sequential and logical
✓ Summary is substantive (not just "This chapter is about X")
✓ Entities are central to the chapter (not just mentioned once)
✓ Keywords are domain-specific (not generic words)
✓ JSON is valid
